🤔 为什么需要反代模式？PROXYIP又是什么？
 
 用最简单的话解释：

 🚀 问题是这样的：
 我们的 ZQ-Edtu 服务部署在 Cloudflare Workers 上。但是 Cloudflare 有个限制——他们的 Workers 无法直接访问 Cloudflare 自己的服务。
这就像名叫 Cloudflare 的小白永远无法从自己的眼睛里看到自己。

 💡 解决方案：
 为了解决这个问题，我们需要借助外部服务器（ProxyIP、SOCKS5 或 HTTP）作为「跳板」，通过这些第三方服务器来代理访问 Cloudflare 的服务。
相当于找了一面（ProxyIP/SOCKS5/HTTP）镜子，这样 Cloudflare 就能通过镜子看到了自己。

 📍 实际效果： 
     
  • 👤 访问非 Cloudflare CDN 的网站时（如油管、谷歌等），你的 IP 由「优选 IP」决定
    •  
  • 🌐 访问由 Cloudflare CDN 托管的网站时（如推特、ChatGPT等），你的 IP 由「反代模式（PROXYIP/SOCKS5/HTTP）」决定
    •  
 
 
 
  
 
 
🤔 为什么需要反代模式？PROXYIP又是什么？
 
❓ 为什么需要反代模式？
 
 根据 Cloudflare Workers 的 TCP Sockets 官方文档 说明，存在以下技术限制： 
 
 ⚠️ Outbound TCP sockets to Cloudflare IP ranges ↗ are temporarily blocked, but will be re-enabled shortly. 
 
 这意味着 Cloudflare Workers 无法直接连接到 Cloudflare 自有的 IP 地址段。为了解决这个限制，需要借助第三方云服务商的服务器作为「跳板」： 
 
 而我们的 ZQ-Edtu 是部署在 Cloudflare Workers 上的代理服务。根据 Cloudflare Workers 的限制政策，Workers 节点无法直接连接到 Cloudflare 自有的 IP 地址段。这意味着： 
 
     
  • ✅ 可以正常访问非 Cloudflare CDN 的站点
    •  
  • ❌ 无法直接访问由 Cloudflare CDN 托管的网站
    •  
 
 因此需要反代模式来解决这个问题。反代模式通过第三方服务器作为「中介」，帮助 Workers 绕过这一限制。 
 
 
 
 
Cloudflare Workers
 
发起请求
 
 
→
 
 
PROXYIP/SOCKS5/HTTP 服务器
 
第三方代理
 
 
→
 
 
Cloudflare 服务
 
目标服务
 
 
 
 通过第三方服务器反向代理 Cloudflare 的 443 端口，实现 Workers 对 Cloudflare 服务的访问 
 
 
💡 实际应用效果
 
 因此，你访问网络时的 IP 归属地取决于访问的目标网站类型： 
 
     
  • 👤 访问非 Cloudflare CDN 站点时（如油管、谷歌等），你的 IP 归属地由「优选 IP」决定
    •  
  • 🌐 访问由 Cloudflare CDN 托管的网站时（如推特、ChatGPT等），你的 IP 归属地由「ProxyIP/SOCKS5/HTTP」决定
    •  
 
 
  
   
  
 
  
 
   
    
  
 
 
 
🔐 什么是 ECH？
 
 ECH（Encrypted Client Hello，加密客户端问候） 是 TLS 的一项新特性，用来 将原本会明文暴露的域名信息一起加密。 
 
 如果你希望了解更完整、偏技术细节的说明，可以参考 Cloudflare 官方博客：
 👉 https://blog.cloudflare.com/zh-cn/announcing-encrypted-client-hello/ 
 
📖 背景说明
 
 当我们使用 WS + TLS 的节点进行代理时： 
 
     
  • ✅ 实际通信内容已经被 TLS 加密，GFW 无法看到你访问的具体内容 
    •  
  • ❌ 但在 TLS 握手阶段，仍然会暴露一个关键信息：SNI 
    •  
 
 SNI 就是节点的伪装域名（HOST） 
 
 也就是说： 
 
 GFW 虽然不知道你在访问什么，但知道你在"扶墙"，并且知道你连接的是哪个域名。 
 
⚠️ 会带来什么问题？
 
 这就是为什么在以下场景中经常出现异常情况： 
 
     
  • ❌ v2rayN 批量测试真链接延迟
    •  
  • ❌ Clash 策略组自动选择节点
    •  
 
 表现通常为： 
 
 所有节点瞬间 -1，全部无法使用 
 
 其原因并不是节点真的失效，而是 运营商 / GFW 通过阻断节点域名的访问来干扰代理连接。
 由于这种阻断大多是自动化策略，容易误判，因此 过一段时间节点又可能恢复正常，反复循环。 
 
💡 ECH 的作用
 
 ECH 的核心作用是： 
 
 将 SNI（也就是节点域名）加密 
 
 启用 ECH 后： 
 
     
  • ✅ GFW 无法获取真实的节点域名 
    •  
  • ✅ 外部观察到的域名将统一显示为：cloudflare-ech.com 
    •  
 
 这从源头上 阻断了通过域名精准封锁节点的手段。 
 
🤔 ECH 会不会被封？
 
 理论上，GFW 只需 直接阻断 cloudflare-ech.com 即可。 
 
 但这样会： 
 
     
  • ❌ 误伤大量正常使用 ECH 的网站和服务
    •  
  • ❌ 带来较高的封锁成本和副作用
    •  
 
 因此，ECH 能持续多久，取决于 GFW 的取舍与策略，至少在目前阶段仍然非常有效。 
 
 
 
 
 
🚀 如何使用 ECH？
 
 使用 ECH 非常简单，只需要 ECH 设置 为 开启 后更新订阅即可。 
 
📱 支持 ECH 的客户端
 
Windows：
 
     
  •  v2rayN ≥ v7.17.0
     https://github.com/2dust/v2rayN/releases 
    •  
 
Android：
 
     
  •  v2rayNG ≥ v2.0.0
     https://github.com/2dust/v2rayNG/releases 
    •  
 
Clash.Meta（mihomo 内核）：
 
     
  •  所有使用 clash.meta / mihomo 内核 的客户端均支持 ECH 
    •  
  •  ⚠️ 需要 DNS 配合使用 
    •  
 
⚠️ Clash 用户注意事项
 
 当前订阅配置中已 内置 ECH 所需的 DNS 设置。 
 
 如果出现以下情况： 
 
     
  • ✅ 已开启 ECH
    •  
  • ✅ 已更新订阅
    •  
  • ❌ ECH 节点仍然无法使用
    •  
 
 请检查： 
 
 是否在更新订阅时覆盖了原有的 DNS 配置 
 
 建议：不要覆盖订阅中自带的 DNS 设置 
 
 
 
 
 
🔍 如何确认 ECH 是否已经生效？
 
 可以通过一个 简单且直观的方法 来验证 ECH 是否正常工作。 
 
📋 验证步骤
 
     
  1. 打开节点的 「⚙️ 详细配置信息」
    2.  
  2. 在 HOST 中手动填写一个 已被墙的域名，例如： *.workers.dev 
    3.  
  3. 保存配置后 重新更新订阅 
    4.  
  4. 在节点列表中查找： 
       
    • - HOST 为 *.workers.dev 的节点
      •  
    • - 并尝试连接该节点
      •  
     
    5.  
 
✅ 如何判断结果？
 
     
  •  ✅ 如果节点可以正常连接使用
     说明 ECH 已成功生效，真实被墙的域名已被隐藏，对外仅显示为 cloudflare-ech.com 
    •  
  •  ❌ 如果节点无法连接
     说明 ECH 未生效，或客户端版本、DNS 配置存在问题 
    •
