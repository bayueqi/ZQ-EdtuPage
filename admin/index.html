🤔为什么需要反代模式？PROXYIP又是什么？
用最简单的话解释：

🚀问题是这样的：
我们的ZQ-Edtu服务部署在Cloudflare Workers上。但是Cloudflare有个限制——他们的Workers无法直接访问Cloudflare自己的服务。
这就像名叫Cloudflare的小白永远无法从自己的眼睛里看到自己。

💡解决方案：
为了解决这个问题，我们需要借助外部服务器（ProxyIP、SOCKS5或HTTP）作为「跳板」，通过这些第三方服务器来代理访问Cloudflare的服务。
相当于找了一面（ProxyIP/SOCKS5/HTTP）镜子，这样Cloudflare就能通过镜子看到了自己。

📍实际效果：
  • 👤访问非Cloudflare CDN的网站时（如油管、谷歌等），你的IP由「优选IP」决定
  • 🌐访问由Cloudflare CDN托管的网站时（如推特、ChatGPT等），你的IP由「反代模式（PROXYIP/SOCKS5/HTTP）」决定
🤔为什么需要反代模式？PROXYIP又是什么？
❓为什么需要反代模式？
根据Cloudflare Workers的TCP Sockets官方文档说明，存在以下技术限制：
⚠️Outbound TCP sockets toCloudflare IP ranges↗are temporarily blocked,but will be re-enabled shortly.
这意味着Cloudflare Workers无法直接连接到Cloudflare自有的IP地址段。为了解决这个限制，需要借助第三方云服务商的服务器作为「跳板」：
而我们的ZQ-Edtu是部署在Cloudflare Workers上的代理服务。根据Cloudflare Workers的限制政策，Workers节点无法直接连接到Cloudflare自有的IP地址段。这意味着：
  • ✅可以正常访问非Cloudflare CDN的站点
  • ❌无法直接访问由Cloudflare CDN托管的网站
因此需要反代模式来解决这个问题。反代模式通过第三方服务器作为「中介」，帮助Workers绕过这一限制。

🔐什么是ECH？
ECH（Encrypted Client Hello，加密客户端问候）是TLS的一项新特性，用来将原本会明文暴露的域名信息一起加密。
如果你希望了解更完整、偏技术细节的说明，可以参考Cloudflare官方博客：
👉https:
📖背景说明
当我们使用WS+TLS的节点进行代理时：
  • ✅实际通信内容已经被TLS加密，GFW无法看到你访问的具体内容
  • ❌但在TLS握手阶段，仍然会暴露一个关键信息：SNI
SNI就是节点的伪装域名（HOST）
也就是说：
GFW虽然不知道你在访问什么，但知道你在"扶墙"，并且知道你连接的是哪个域名。
⚠️会带来什么问题？
这就是为什么在以下场景中经常出现异常情况：
  • ❌v2rayN批量测试真链接延迟
  • ❌Clash策略组自动选择节点
表现通常为：
所有节点瞬间-1，全部无法使用
其原因并不是节点真的失效，而是运营商/GFW通过阻断节点域名的访问来干扰代理连接。
由于这种阻断大多是自动化策略，容易误判，因此过一段时间节点又可能恢复正常，反复循环。
💡ECH的作用
ECH的核心作用是：
将SNI（也就是节点域名）加密
启用ECH后：
  • ✅GFW无法获取真实的节点域名
  • ✅外部观察到的域名将统一显示为：cloudflare-ech.com
这从源头上阻断了通过域名精准封锁节点的手段。
🤔ECH会不会被封？
理论上，GFW只需直接阻断cloudflare-ech.com即可。
但这样会：
  • ❌误伤大量正常使用ECH的网站和服务
  • ❌带来较高的封锁成本和副作用
因此，ECH能持续多久，取决于GFW的取舍与策略，至少在目前阶段仍然非常有效。

🔍如何确认ECH是否已经生效？
可以通过一个简单且直观的方法来验证ECH是否正常工作。
📋验证步骤
  1. 打开节点的「⚙️详细配置信息」
  2. 在HOST中手动填写一个已被墙的域名，例如：*.workers.dev
  3. 保存配置后重新更新订阅
  4. 在节点列表中查找：
    • -HOST为*.workers.dev的节点
    • -并尝试连接该节点
✅如何判断结果？
  • ✅如果节点可以正常连接使用
    说明ECH已成功生效，真实被墙的域名已被隐藏，对外仅显示为cloudflare-ech.com
  • ❌如果节点无法连接
    说明ECH未生效，或客户端版本、DNS配置存在问题
